locking/mutex: Support Clang's context analysis
31 messages 4 participants
Marco Elver, Bart Van Assche, Peter Zijlstra, Linus Torvalds
AI Summary

This patch series adds support for Clang's thread safety analysis (also called "context analysis" or "capability analysis") to the kernel's mutex implementation. The goal is to enable compile-time verification that locks are properly acquired before accessing protected data and properly released afterward.

  • Problem: Runtime lockdep is excellent but has overhead; static analysis can catch bugs at compile time with zero runtime cost
  • Solution: Add annotations like __acquires(), __releases(), __must_hold() that Clang's analyzer understands
  • Challenge: Sparse already uses similar annotations but with different semantics; this work removes Sparse support in favor of Clang
Outcome: Active development, part of larger context analysis infrastructure
cleanup: Basic compatibility with context analysis
31 messages 3 participants
Marco Elver, Bart Van Assche, Boqun Feng
AI Summary

Preparatory cleanup patches to make the kernel codebase compatible with Clang's context analysis. The kernel has many "workarounds" and "hacks" added over the years to satisfy Sparse's context tracking, which work differently from Clang's approach.

  • Removes or adjusts annotations that were added specifically to "fake out sparse"
  • Standardizes locking annotation patterns across subsystems
  • Prepares guard macros (like DEFINE_LOCK_GUARD) to work with new analysis
Outcome: Foundation for enabling static locking verification
srcu: Support Clang's context analysis
31 messages 3 participants
Marco Elver, Bart Van Assche, Linus Torvalds
AI Summary

Extends context analysis support to SRCU (Sleepable Read-Copy-Update), a variant of RCU that allows sleeping in read-side critical sections. This is particularly tricky because SRCU has more complex semantics than regular mutexes.

  • Adds __srcu_read_lock_must_hold() helper for compile-time verification
  • Bart Van Assche raised concerns: the new helper requires SRCU reader lock unconditionally, but some code legitimately uses srcu_dereference_check() with alternative conditions
  • Debate about whether this forces developers to add suppression annotations
Outcome: Under discussion, API design trade-offs being evaluated
kbuild: remove gcc's -Wtype-limits
60 messages 6 participants
AI Summary

Extended discussion about removing GCC's -Wtype-limits warning from the kernel build. This warning fires when comparing values against type limits (e.g., if (unsigned_val < 0)), but produces many false positives in generic/macro code.

  • Problem: Generic macros like min()/max() trigger warnings when instantiated with different types
  • Debate: Some argue warnings catch real bugs; others say noise drowns signal
  • Related to overflow helper changes (removing is_non_negative()/is_negative())
Outcome: Active discussion on warning policy

compiler-context-analysis: Add infrastructure for Context Analysis with Clang
30 messages
AI Summary

This patch introduces the core infrastructure for using Clang's thread safety analysis in the Linux kernel. The key insight is that Clang's "capability analysis" can statically verify locking correctness at compile time, complementing the runtime lockdep checker.

  • Creates include/linux/compiler-context-analysis.h with new annotation macros
  • Defines "context locks" - abstract locks that Clang tracks but don't necessarily correspond to real lock objects
  • Supports both exclusive and shared (reader/writer) lock semantics
  • Key decision: Removes Sparse's context tracking support entirely in favor of Clang
Outcome: Foundation for kernel-wide static lock checking

25 messages
AI Summary

Proposal to add a new is_const() macro as a cleaner replacement for the existing __is_constexpr() hack. The kernel needs to detect at compile time whether an expression is a constant.

  • Use case: Macros like BUILD_BUG_ON() need different implementations for constant vs runtime values
  • Current approach: __is_constexpr() uses a sizeof trick with ternary operator and array types
  • Cleaner API proposed to improve readability and maintainability
Outcome: Under discussion for improved constant detection

AI Summary

  • Sparse impact: PIE code generates different symbol types that sparse needs to understand
  • Part of 28-patch series touching architecture-specific code

overflow: Introduce overflows_type() and castable_to_type()
80 messages 7 participants
AI Summary

Major addition to the kernel's overflow checking infrastructure, led by Kees Cook (Chrome OS security). This was the largest discussion thread of 2022, spanning multiple patch revisions (v11+).

  • overflows_type(n, T): Tests if value n would overflow when stored in type T. Works as constant expression for static_assert()
  • __castable_to_type(n, T): Like __same_type() but allows constants that fit in the target type
  • Manually constructed since __builtin_add_overflow() isn't a constant expression
  • Extensive KUnit test suite added with 391+ test cases covering all integer type combinations
Outcome: Merged, now used for compile-time overflow detection

Introduce and use absolute_pointer macro
24 messages
AI Summary

Introduction of the absolute_pointer() macro to handle pointers to fixed/absolute memory addresses without triggering compiler warnings or undefined behavior concerns.

  • Problem: Code like strcmp((char *)0x1000, "test") triggers warnings about comparing with NULL or using absolute addresses
  • Solution: absolute_pointer(x) macro that tells the compiler this is an intentional absolute address, not a NULL pointer bug
  • Used primarily on Alpha architecture and VirtualBox shared folder driver
  • Implemented using volatile pointer casts to prevent compiler optimizations
Outcome: Merged into compiler.h, fixes sparse and GCC warnings

pragma once: treewide conversion
35 messages
AI Summary

Alexey Dobriyan proposed converting all kernel headers from traditional #ifndef/#define/#endif include guards to #pragma once. Linus Torvalds tested and rejected the proposal.

  • Proposal benefits: Cleaner headers, no risk of guard name collisions, slightly faster compilation
  • Sparse problem: Sparse uses pathname equality for #pragma once, so #include "header.h" vs #include "./header.h" are treated as different files, causing infinite recursion
  • GCC behavior: Even GCC opened and read files multiple times with different path spellings
  • Linus's verdict: "#pragma once is complete garbage... fundamentally more complicated than the #ifdef guard, and it has absolutely zero upsides"
Outcome: Rejected - traditional include guards remain
